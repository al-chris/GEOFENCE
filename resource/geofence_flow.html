Geofence Fix Flow

Sequence Diagram
sequenceDiagram participant G as "GNSS Receiver" participant D as "Serial Driver / gpsd" participant P as "NMEA Parser" participant R as "ROS2 Publisher" participant F as "GeoFence Node" participant HW as "GPIO Hardware" participant U as "User / Operator" note right of G: Satellites (GPS/GLONASS/BeiDou/Galileo) G->>D: raw NMEA / UBX frames (UART / USB / TCP) D->>P: deliver raw sentences/frames P->>R: publish sensor_msgs/NavSatFix on /gps/fix R->>F: NavSatFix -> gps_callback(lat,lon,status) F->>F: Kalman filter init/update (filter noisy fixes) F->>F: point-in-polygon check (shapely, lon/lat order) alt inside F->>HW: green ON; red OFF; buzzer OFF F-->>U: log "inside" else outside F->>HW: red ON; buzzer ON; green OFF F->>R: publish Twist on /cmd_vel (stop) F-->>U: log "BOUNDARY CROSSED" end note left of R: NavSatFix includes latitude, longitude, status, covariance note over G: optional RTK corrections (NTRIP/base station) note over P: simulated publishers (e.g., mock_gps_publisher) provide test fixes

This diagram shows how a GPS "fix" flows from the GNSS receiver through drivers/parsers into ROS as a `NavSatFix`, then into the `geofence_node` which applies filtering and polygon checks before changing GPIO or publishing stop commands.
